# Page Archetypes

Page archetypes govern conceptual page planning before implementation. Choose one dominant archetype, note exceptions, and preserve action priority and scroll ownership.

## Scope

These eight archetypes are conceptual page-planning patterns.

## Archetype Selection

Select the archetype that matches the page job, region topology, primary action, and scroll owner. Mixed pages must choose a dominant archetype and document exceptions.

## Single Workspace

- Page job: complete one focused work activity
- Region topology: explicit regions with one clear owner per region.
- Primary action: one dominant action remains available at the right breakpoint.
- Secondary action pattern: secondary actions stay contextual and do not compete with the primary action.
- Desktop behavior: desktop may expose simultaneous regions when comparison helps.
- Tablet transformation: tablet reduces simultaneous regions while preserving action access.
- Mobile transformation: mobile uses one primary sequence and avoids competing scroll owners.
- Likely scroll owner: page or named primary content region.
- State responsibilities: loading, empty, error, disabled, success, focus, and reduced-motion states.
- Inappropriate usage: do not use this archetype when it obscures the page job or creates decorative structure.

## Responsive Transformation Principles

Responsive changes must preserve job, action priority, reading order, focus order, and scroll ownership. Do not create different products at different breakpoints.

## State Responsibilities

Each archetype must define loading, empty, error, disabled, success, focus, and reduced-motion states when those states can occur.

## Inappropriate Usage

Do not choose an archetype because it looks fashionable. Choose it because the page job and region topology require it.
